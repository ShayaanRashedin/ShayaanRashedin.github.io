Conducting a Security Audit

1. Organization Background & Audit Scope

Botium Toys is a retail company selling toys online and in physical stores. They manage a mix of digital and physical assets—employee workstations, on-premises servers, an internal network, plus some legacy infrastructure.
Customer payments (credit cards) and sensitive personal data (PII/SPII) are processed and stored internally.
This audit evaluates current security controls and compliance with PCI DSS, GDPR, and SOC 2 standards.

Scope: Entire security program—technology, physical security, and processes.
Goals: Assess controls, identify compliance gaps, and recommend steps to reduce risk and strengthen posture.

2. Controls Assessment Checklist

Control Status Notes
Least Privilege ❌ All employees have unrestricted access to sensitive data; implement role-based restrictions.
Disaster Recovery Plan ❌ No formal DRP exists; develop and test a comprehensive plan.
Password Policies ❌ Weak standards; enforce complexity, rotation, and lockout policies.
Separation of Duties ❌ Critical functions handled by one person; split responsibilities to reduce fraud/error.
Firewall ✅ Properly configured to control inbound/outbound traffic.
Intrusion Detection System (IDS) ❌ Missing; install and tune an IDS to alert on suspicious activity.
Backups ❌ No regular backups; schedule automated backups to prevent data loss.
Antivirus Software ✅ Up-to-date AV is deployed and monitored.
Legacy System Monitoring ❌ No maintenance plan; document and review legacy platforms regularly.
Encryption ❌ Data at rest/in transit is unencrypted; apply full-disk and data-level encryption.
Password Management System ❌ No centralized vault; adopt a secure password manager.
Physical Locks ✅ Facility access controlled by locks.
CCTV Surveillance ✅ Cameras operational for deterrence and forensics.
Fire Safety Systems ✅ Alarms and suppression systems functional.

3. Compliance Checklist

Standard Requirement Status Notes
PCI DSS Authorized Access to Card Data ❌ All employees can view cardholder data; restrict by role.
PCI DSS Secure Storage & Transmission ❌ No encryption or controls on stored card data.
Encryption Procedures All Transaction Touchpoints ✅ Recommendation to implement encryption company-wide.
Password Management Policies Strong Handling & Renewal ✅ Adopt secure password handling and renewal policies.
GDPR EU Data Privacy ❌ PII stored unencrypted; apply strict access and encryption.
GDPR Breach Notification ✅ Plan exists to notify within 72 hours of a breach.
Data Classification Asset Inventory & Class Scheme ❌ No formal classification; create an inventory and scheme.
Privacy Policies Enforcement Across Departments ✅ Policies in place and enforced.
User Access Policies Role-Based Access ❌ No formal access policy; establish and enforce policies.
Data Integrity Accuracy & Non-Repudiation ✅ Controls ensure data remains accurate and unaltered.
Data Availability Authorized Access Only ❌ Universal access currently; limit to authorized roles.

4. Recommendations

  • Implement role-based access control (RBAC) and enforce least privilege.
  • Deploy a centralized password management solution with strong policies.
  • Introduce full-disk and data-level encryption for all sensitive data.
  • Develop, test, and regularly review a disaster recovery plan (DRP) and backups.
  • Install and configure an Intrusion Detection System (IDS) with alerting.
  • Document and schedule maintenance reviews for all legacy systems.
  • Enforce separation of duties for critical business functions.
  • Create a formal data classification and inventory program.
  • Expand security awareness training, covering PCI DSS, GDPR, and SOC 2.
Analyzing Network Structure and Security

Incident Report Analysis

Summary

Your organization experienced a Distributed Denial of Service (DDoS) attack targeting its internal network through a flood of ICMP packets. The vulnerability was due to an unconfigured firewall which allowed the attack to overwhelm network resources, causing a two-hour outage. Remediation efforts included temporarily blocking ICMP packets and implementing new firewall rules, IP source verification, and traffic monitoring systems.

Identify

  • Conducted internal audits and found an unconfigured firewall as the primary vulnerability.
  • Identified lack of ICMP traffic rate-limiting as a major risk.
  • Recognized that current network infrastructure lacked adequate traffic anomaly detection tools.
  • Mapped out affected systems and potential entry points for malicious actors.

Protect

  • Implemented firewall rules to limit the rate of incoming ICMP packets.
  • Enabled source IP verification to defend against spoofed packets.
  • Developed updated network access policies.
  • Trained staff on identifying DDoS-related symptoms and emergency procedures.

Detect

  • Installed and configured network monitoring software to alert for abnormal traffic patterns.
  • Deployed IDS/IPS to flag suspicious ICMP traffic and provide better visibility.
  • Enhanced logging and alerting to identify unusual spikes in packet traffic.

Respond

  • Immediately blocked incoming ICMP traffic to contain the incident.
  • Worked with the incident response team to restore critical services quickly.
  • Investigated source and nature of attack using logs and traffic analysis.
  • Documented the event for future reference and used it as a learning opportunity.

Recover

  • Gradually restored all network services once the threat was neutralized.
  • Validated firewall configurations to ensure lasting protection.
  • Updated incident response and recovery plans based on lessons learned.
  • Communicated with stakeholders regarding the resolution and prevention measures.

Reflections/Notes

This event highlighted the importance of basic firewall configurations and proactive network traffic monitoring. The use of the NIST Cybersecurity Framework helped systematically improve our organization’s defense and response capabilities. Regular reviews of firewall rules and detection systems are now scheduled to avoid future oversights.

Using Linux Commands to Manage File Permissions

Current File Permissions

This document displays the file structure of the /home/researcher2/projects directory and the permissions of the files and subdirectory it contains.

In the /home/researcher2/projects directory, there are five files with the following permissions:

  • project_k.txt: User = read, write; Group = read, write; Other = read, write
  • project_m.txt: User = read, write; Group = read; Other = none
  • project_r.txt: User = read, write; Group = read, write; Other = read
  • project_t.txt: User = read, write; Group = read, write; Other = read
  • .project_x.txt: User = read, write; Group = write; Other = none

There is also one subdirectory named drafts with permissions:

  • User = read, write, execute
  • Group = execute
  • Other = none

Project Description

In this activity, I acted as a security professional managing file and directory permissions for a research team. My main goal was to analyze current permissions, determine if they matched authorization policies, and use Linux commands to correct any discrepancies. This ensured that only the appropriate users could access or modify specific files and directories.

Check File and Directory Details

To inspect the current file and directory permissions, I used the following command:

ls -la /home/researcher2/projects

This lists all files (including hidden), their permission strings, ownerships, and other details, helping identify which needed adjustment.

Describe the Permissions String

One example string is:

-rw-rw-r-- 1 researcher2 researcher2 1234 project_k.txt

This breaks down as:

  • -: Regular file
  • rw-: User (owner) has read/write
  • rw-: Group has read/write
  • r--: Others have read only

Change File Permissions

The organization prohibits write for "others." To correct project_k.txt:

Command used:

chmod o-w /home/researcher2/projects/project_k.txt

Result:

-rw-rw-r-- 1 researcher2 researcher2 1234 project_k.txt

Change Permissions on Hidden File

For .project_x.txt, only user and group should read:

Command used:

chmod 440 /home/researcher2/projects/.project_x.txt

Result:

-r--r----- 1 researcher2 researcher2 987 .project_x.txt

Change Directory Permissions

To restrict drafts to researcher2 only:

Command used:

chmod 700 /home/researcher2/projects/drafts

Result:

drwx------ 2 researcher2 researcher2 4096 drafts

Summary

In this exercise, I checked file and directory permissions using ls -la, interpreted the permission strings, and modified access controls with chmod. I ensured only authorized users had appropriate rights, revoking unnecessary permissions. This aligned the system with secure access policies and supported organizational data protection efforts.

Applying Filters to SQL Queries

Project Description

In this project, I acted as a security professional investigating potential security threats within a large organization. I used SQL queries with filtering conditions (AND, OR, NOT, LIKE) to analyze login attempts and employee data. The goal was to identify unusual login activity and update machine security for employees in specific departments. Through this process, I demonstrated the ability to filter datasets efficiently to identify potential threats.

Retrieve After-Hours Failed Login Attempts

SELECT *
        FROM log_in_attempts
        WHERE login_time > '18:00'
        AND (success = 0 OR success = FALSE);

Retrieve Login Attempts on Specific Dates

SELECT *
        FROM log_in_attempts
        WHERE login_date = '2022-05-09'
        OR login_date = '2022-05-08';

Retrieve Login Attempts Outside of Mexico

SELECT *
        FROM log_in_attempts
        WHERE country NOT LIKE 'MEX%'
        AND country NOT LIKE 'Mexico%';

Retrieve Employees in Marketing

SELECT *
        FROM employees
        WHERE department = 'Marketing'
        AND office LIKE 'East%';

Retrieve Employees in Finance or Sales

SELECT *
        FROM employees
        WHERE department = 'Finance'
        OR department = 'Sales';

Retrieve All Employees Not in IT

SELECT *
        FROM employees
        WHERE department != 'Information Technology';

Summary

In this project, I applied SQL filtering techniques to identify login anomalies and assist in machine updates for specific employee groups. Using a variety of operators and the LIKE keyword, I successfully queried relevant data to support security investigations. This exercise demonstrated my proficiency in filtering datasets and using SQL to address real-world cybersecurity issues.

Identifying Vulnerabilities for a Small Business

Vulnerability Assessment Report

Date: July 10, 2025

System Description

The server hardware consists of a powerful CPU and 128 GB of memory. It runs the latest Linux OS and hosts a MySQL database. It uses IPv4 networking, communicates with internal servers, supports global remote access, and secures traffic with SSL/TLS. The server has been publicly accessible since launch.

Scope

This assessment covers the confidentiality, integrity, and availability of data on the public database server. It does not include physical security or unrelated IT systems. Conducted over three months per NIST SP 800-30 Rev. 1.

Purpose

The database holds customer and business data critical for operations. A breach could cause legal liability, reputational damage, and financial loss. This report identifies vulnerabilities and recommends mitigations.

Risk Assessment

Threat Source Threat Event Likelihood Severity Risk (L×S)
Cybercriminals Exfiltrate customer data via SQL injection 3 3 9
Insider (employee) Misuse credentials to access restricted data 2 3 6
Competitor Launch DDoS attack to disrupt operations 2 2 4

Approach

This qualitative assessment used the NIST SP 800-30 framework to score threats based on public accessibility and authentication reliance. Likelihood and severity reflect attacker capability and impact. Focus is on cyber-based vectors.

Remediation Strategy

  • Implement Principle of Least Privilege so employees only access necessary data.
  • Enforce Multi-Factor Authentication (MFA) to block stolen-credential access.
  • Restrict the server behind a VPN or firewall to remove public exposure.
  • Deploy monitoring and logging tools to detect anomalies and support response.

Importing and Parsing a Text File in a Security-Related Scenario

Project Description

You are a security professional at a healthcare company, tasked with maintaining a list of employee IP addresses that are permitted to access a restricted subnetwork. This list, stored in allow_list.txt, must be regularly updated to remove any IPs identified in a separate remove_list that denotes revoked access. The goal of the algorithm is to identify and remove any IPs listed in the remove list from the allow list file using Python.

Open the File in Read Mode

with open("allow_list.txt", "r") as file:
            ip_addresses = file.read()

Split Contents into a List

ip_addresses = ip_addresses.split()

Iterate Through the Remove List

remove_list = ["192.168.1.1", "10.0.0.5"]
        for ip in remove_list:
            if ip in ip_addresses:
                ip_addresses.remove(ip)

Write the Updated List Back to the File

with open("allow_list.txt", "w") as file:
            file.write("\n".join(ip_addresses))

Summary

This Python algorithm automates the upkeep of an IP allow-list by:

  1. Reading the existing list from allow_list.txt.
  2. Parsing it into a Python list for easy manipulation.
  3. Removing any addresses found in a predefined remove_list.
  4. Writing the cleaned list back to the original file.
By automating these steps, it ensures that only currently authorized IPs retain access, strengthening network security through up-to-date access controls.
