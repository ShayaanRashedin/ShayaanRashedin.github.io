Analyzing Network Structure and Security

Project Overview

I acted as a cybersecurity analyst at a multimedia company that offers web, graphic, and social media services. After the company experienced a distributed denial of service attack that sent a large volume of ICMP packets through an unconfigured firewall and brought internal services offline for two hours, I developed a security improvement plan based on the NIST Cybersecurity Framework. I focused on firewall misconfigurations, DDoS response, and long term network hardening to prevent similar attacks in the future.

Implemented Controls

  • Added a firewall rule to rate limit incoming ICMP packets.
  • Enabled source IP verification on the firewall to prevent spoofed addresses.
  • Deployed network monitoring software to detect abnormal traffic patterns.
  • Installed and tuned an IDS and IPS to filter suspicious ICMP traffic.

Network Security Improvement Plan Based on the NIST Cybersecurity Framework

Identify

Conduct regular audits of networks, devices, and access privileges to uncover misconfigurations such as unconfigured firewall rules. Perform vulnerability scans on firewalls and network equipment.

Protect

Enforce security policies that require proper firewall configuration, protocol rate limiting, source IP validation, and network segmentation. Train staff on incident response and firewall management.

Detect

Deploy network monitoring and anomaly detection tools to alert on traffic spikes. Configure IDS and IPS signatures and thresholds to catch DDoS behavior before service disruption.

Respond

Develop a DDoS specific incident response playbook that defines roles, communications, and escalation procedures. Conduct tabletop exercises to validate readiness.

Recover

Maintain backups of critical network configurations and system states. Establish rapid service restoration procedures and perform post incident reviews to update controls and lessons learned.

Identifying Vulnerabilities for a Small Business

Project Overview

I conducted a hands on vulnerability assessment of a publicly accessible Linux server that hosts a MySQL database for a fictional ecommerce organization. The database server is critical for the business and has been exposed to the internet for three years with no inbound firewall restrictions. It runs Linux, allows remote employee access worldwide, communicates with internal systems over IPv4, and uses SSL and TLS to protect data in transit. Using the NIST SP 800 30 framework, I analyzed threats, estimated the likelihood and impact of different attack scenarios, and proposed remediation strategies to protect confidentiality, integrity, and availability.

System Description

The server uses a high performance CPU, 128 GB RAM, the latest Linux operating system, and MySQL. It supports global remote access with SSL and TLS and it is directly reachable from the internet with no firewall access control lists limiting inbound database traffic.

Scope

This assessment focuses on protecting customer data on the exposed database server. It does not cover physical security or other infrastructure. The three month evaluation followed NIST SP 800 30 guidance.

Purpose

The database is essential for sales and marketing. A breach could result in data theft, downtime, legal exposure, and reputational damage. My goal was to identify vulnerabilities before they could be exploited and to recommend effective defensive controls.

Risk Assessment

Threat Source Threat Event Likelihood Severity Risk (L × S)
Cybercriminals Exfiltrate customer data using SQL injection 3 3 9
Insider employee Misuse credentials to access restricted data 2 3 6
Competitor Launch a DDoS attack to disrupt operations 2 2 4

Approach

Following NIST SP 800 30 Revision 1, I scored threats based on attacker capability and system exposure. I estimated likelihood and severity using threat actor profiles and the potential business impact. I focused mainly on cyber based attack vectors because the server is publicly exposed.

Remediation Strategy

  • Implement the Principle of Least Privilege to restrict data access by role.
  • Enforce Multi Factor Authentication for all database accounts.
  • Restrict server access behind a VPN or strict firewall access control lists.
  • Deploy continuous monitoring and perform regular vulnerability scans.

Compliance Checklist

Using Linux Commands to Manage File Permissions

Project Overview

I acted as a security analyst supporting a research team in a large organization. I audited and corrected file and directory permissions within a Linux environment to make sure that users only had the appropriate level of access that matched organizational security policies. I inspected file permissions in the /home/researcher2/projects directory, identified unnecessary or unsafe permissions such as public write access, and used Linux commands to fix them so that only authorized users retained access.

Check File and Directory Details

To inspect the current file and directory permissions, I used:

ls -la /home/researcher2/projects

Describe the Permissions String

Example permissions string:

-rw-rw-r-- 1 researcher2 researcher2 1234 project_k.txt
  • Regular file
  • User has read and write
  • Group has read and write
  • Others have read only

Change File Permissions

The organization prohibits write access for others. To correct project_k.txt:

chmod o-w /home/researcher2/projects/project_k.txt

Result:

-rw-rw-r-- 1 researcher2 researcher2 1234 project_k.txt

Change Permissions on a Hidden File

For .project_x.txt:

chmod 440 /home/researcher2/projects/.project_x.txt

Result:

-r--r----- 1 researcher2 researcher2 987 .project_x.txt

Change Directory Permissions

To restrict the drafts directory to researcher2 only:

chmod 700 /home/researcher2/projects/drafts

Result:

drwx------ 2 researcher2 researcher2 4096 drafts

Summary

I checked file and directory permissions using ls -la, interpreted the permissions string, and modified access controls using chmod. I ensured that only authorized users had appropriate rights and I removed unnecessary permissions for others. This aligned the system with secure access policies and supported organizational data protection efforts.

Importing and Parsing a Text File in a Security Related Scenario

Project Overview

I acted as a security professional at a fictional healthcare company and was responsible for maintaining a list of employee IP addresses that are allowed to access a restricted subnetwork. The list is stored in a file named allow_list.txt and must be kept up to date by removing any IP addresses that appear in a separate list of revoked addresses. I wrote a Python script that opens and parses the allow list, removes any addresses that appear in the revoke list, and then writes the cleaned list back to the file so that only currently authorized IPs keep access.

Open and Read the File

with open("allow_list.txt", "r") as file:
    ip_addresses = file.read().split()

Iterate and Remove

remove_list = ["192.168.1.1", "10.0.0.5"]
for ip in remove_list:
    if ip in ip_addresses:
        ip_addresses.remove(ip)

Write the Updated List

with open("allow_list.txt", "w") as file:
    file.write("\n".join(ip_addresses))

Summary

This Python workflow automates access control by reading the existing allow list file, parsing it into a list, removing any revoked IP addresses, and writing the updated list back to disk. The result is a self updating allow list that reflects current authorization policy.

Project Overview

I acted as the incident handler for a simulated ransomware attack against a small United States healthcare clinic. The attack began when a staff member opened a phishing email with a malicious attachment that deployed ransomware and encrypted critical medical and administrative files. I documented the event in an incident handler journal using an investigative framework based on the Five Ws Who, What, When, Where, and Why to capture key details, assess impact, and recommend response and recovery actions.

Incident Handler Journal Entry

Date: July 26, 2025

Description

A ransomware incident occurred at a small United States healthcare clinic following a phishing email. This entry records the event and its key elements.

Tools Used

No specific cybersecurity tools were mentioned in the scenario. Tooling had not yet been deployed.

The Five Ws

  • Who: An organized group of unethical hackers targeting healthcare and transportation sectors.
  • What: Employees received phishing emails with malicious attachments that encrypted critical files and displayed a ransom note.
  • When: Tuesday morning at approximately 9:00 a.m.
  • Where: A small healthcare clinic in the United States that provides primary care services.

Additional Notes

Immediate containment and recovery measures are required. The organization should notify authorities, begin forensic investigation, and implement stronger phishing protections. User awareness training and a reliable data backup strategy should also be prioritized.

Applying Filters to SQL Queries

Project Overview

I investigated potential security threats within a fictional corporate environment by querying employee data and login activity data. I used SQL to detect anomalies such as failed login attempts outside of business hours, activity from non domestic locations, and device update needs by department. This work emphasized the use of AND, OR, NOT, and LIKE operators for real world threat detection.

Retrieve After Hours Failed Login Attempts

SELECT *
FROM log_in_attempts
WHERE login_time > '18:00'
  AND (success = 0 OR success = FALSE);

Retrieve Login Attempts on Specific Dates

SELECT *
FROM log_in_attempts
WHERE login_date = '2022-05-09'
   OR login_date = '2022-05-08';

Retrieve Login Attempts Outside of Mexico

SELECT *
FROM log_in_attempts
WHERE country NOT LIKE 'MEX%'
  AND country NOT LIKE 'Mexico%';

Retrieve Employees in Marketing

SELECT *
FROM employees
WHERE department = 'Marketing'
  AND office LIKE 'East%';

Retrieve Employees in Finance or Sales

SELECT *
FROM employees
WHERE department = 'Finance'
   OR department = 'Sales';

Retrieve All Employees Not in IT

SELECT *
FROM employees
WHERE department != 'Information Technology';

Summary

This activity demonstrates my ability to apply SQL filtering techniques, including logical operators, date and time conditions, and pattern matching, to identify security relevant anomalies in login and employee data.
