Conducting a Security Audit

Project Description

This cybersecurity audit simulated a real-world risk and compliance assessment for Botium Toys, a fictional retail company handling storefront sales and e-commerce with sensitive customer data. I evaluated existing controls, identified gaps against PCI DSS, GDPR, and SOC 2, and provided prioritized recommendations to strengthen security posture.

Scenario

Botium Toys operates both a physical storefront and an online platform, managing PII and credit card data. Their infrastructure includes on-premises servers, employee workstations, legacy hardware, and third-party software. The audit scope covered technology, processes, and policies related to data protection and regulatory compliance.

Controls Assessment Checklist

Control Status Explanation
Least Privilege ❌ All employees have unrestricted access; implement role-based restrictions.
Disaster Recovery Plan ❌ No formal DRP; develop and test a comprehensive plan.
Password Policies ❌ Weak standards; enforce complexity, rotation, and lockout.
Separation of Duties ❌ Critical functions handled by one person; split responsibilities.
Firewall ✅ Properly configured to control inbound/outbound traffic.
Intrusion Detection System (IDS) ❌ Missing IDS; install and tune alerts for suspicious activity.
Backups ❌ No regular backups; schedule automated backups.
Antivirus Software ✅ Up-to-date AV deployed and monitored.
Legacy System Monitoring ❌ No structured maintenance; document and review legacy platforms.
Encryption ❌ Data at rest/in transit is unencrypted; apply full-disk & data-level encryption.
Password Management System ❌ No centralized vault; adopt a secure password manager.
Physical Locks ✅ Facility access controlled by locks.
CCTV Surveillance ✅ Cameras operational for deterrence and forensics.
Fire Safety Systems ✅ Alarms and suppression systems functional.

Compliance Checklist

Requirement Status Explanation
Authorized access to card data ❌ All employees can view cardholder data; restrict by role.
Secure storage & transmission ❌ Card data stored without encryption.
Encryption procedures ✅ Recommend implementing encryption company-wide.
Password management policies ✅ Adopt complexity and renewal policies.
GDPR data privacy ❌ PII stored unencrypted; enforce strict access controls.
GDPR breach notification ✅ Plan exists to notify within 72 hours.
Data classification ❌ No formal classification; implement asset inventory & class scheme.
Privacy policy enforcement ✅ Policies in place and applied across departments.
User access policies ❌ No formal access policy; establish and enforce role-based rules.
Data integrity ✅ Controls ensure data remains accurate and unaltered.
Data availability ❌ Universal access currently; limit to authorized roles.

Recommendations

  • Implement role-based access control (RBAC) and enforce least privilege.
  • Deploy a centralized password management solution.
  • Introduce full-disk and data-level encryption.
  • Develop, test, and regularly review a DRP and backup schedule.
  • Install and configure an IDS with alerting.
  • Maintain legacy systems with formal documentation and review.
  • Enforce separation of duties for critical functions.
  • Create a formal data classification and inventory program.
  • Continue privacy & security training for PCI DSS, GDPR & SOC 2 compliance.
  • Expand security awareness training to mitigate phishing risks.
Analyzing Network Structure and Security

Project Description

In this project, I developed a network security improvement plan based on the NIST Cybersecurity Framework to harden a fictional multimedia company’s internal network after a DDoS attack compromised services for two hours.

Scenario

I acted as a cybersecurity analyst at a multimedia firm offering web, graphic, and social‐media services. The company suffered a DDoS attack via a flood of ICMP packets through an unconfigured firewall, bringing the internal network offline for two hours. The incident response team blocked ICMP traffic, halted non‐critical services, and restored essential functions.

Implemented Controls

  • Added a firewall rule to rate-limit incoming ICMP packets.
  • Enabled source IP verification on the firewall to prevent spoofed addresses.
  • Deployed network monitoring software to detect abnormal traffic patterns.
  • Installed and tuned an IDS/IPS to filter suspicious ICMP traffic.

Network Security Improvement Plan (NIST CSF)

Identify

Conduct regular audits of networks, devices, and access privileges to uncover misconfigurations (e.g., unconfigured firewall rules). Perform vulnerability scans on firewalls and network equipment.

Protect

Enforce security policies ensuring proper firewall configuration, protocol rate limiting, source IP validation, and network segmentation. Train staff on incident response and firewall management.

Detect

Deploy network monitoring and anomaly-detection tools to alert on traffic spikes. Configure IDS/IPS signatures and thresholds to catch DDoS behavior before service disruption.

Respond

Develop a DDoS-specific incident response playbook outlining roles, communications, and escalation procedures. Conduct tabletop exercises to validate readiness.

Recover

Maintain backups of critical network configurations and system states. Establish rapid service restoration procedures and perform post-incident reviews to update controls and lessons learned.

Using Linux Commands to Manage File Permissions

Project Description

In this activity, I acted as a security professional managing file and directory permissions for a fictional research team. My main goal was to analyze current permissions, determine if they matched authorization policies, and use Linux commands to correct any discrepancies. This ensured that only the appropriate users could access or modify specific files and directories.

Scenario

As a Security Analyst working with a research team in a large organization, I was tasked with auditing and correcting file and directory permissions within a Linux environment. The objective was to ensure users had only the appropriate level of access, in alignment with organizational security policies. This activity involved analyzing permission settings on files and directories within the /home/researcher2/projects directory. Based on the assessment, I used Linux commands to correct any misconfigurations—specifically focusing on removing unnecessary write access, securing hidden files, and ensuring restricted directories were only accessible to authorized users.

Check File and Directory Details

To inspect the current file and directory permissions, I used:

ls -la /home/researcher2/projects

Describe the Permissions String

Example permissions string:

-rw-rw-r-- 1 researcher2 researcher2 1234 project_k.txt
  • -: Regular file
  • rw-: User has read & write
  • rw-: Group has read & write
  • r--: Others have read only

Change File Permissions

The organization prohibits write access for “others.” To correct project_k.txt:

chmod o-w /home/researcher2/projects/project_k.txt

Result:

-rw-rw-r-- 1 researcher2 researcher2 1234 project_k.txt

Change Permissions on a Hidden File

For .project_x.txt:

chmod 440 /home/researcher2/projects/.project_x.txt

Result:

-r--r----- 1 researcher2 researcher2 987 .project_x.txt

Change Directory Permissions

To restrict the drafts directory to researcher2 only:

chmod 700 /home/researcher2/projects/drafts

Result:

drwx------ 2 researcher2 researcher2 4096 drafts

Summary

In this exercise, I checked file and directory permissions using ls -la, interpreted the permissions string, and modified access controls with chmod. I ensured only authorized users had appropriate rights, revoking unnecessary permissions for others. This aligned the system with secure access policies, supporting organizational data protection efforts.

Applying Filters to SQL Queries

Project Description

In this project, I investigated potential security threats within a fictional corporate environment by querying employee and login activity data. Through SQL, I applied logical filters to detect anomalies such as failed login attempts outside business hours, activity from non-domestic locations, and device update needs by department. This exercise emphasized the use of AND, OR, NOT, and LIKE operators for real-world threat detection.

Scenario

As a security professional at a large organization, I was tasked with analyzing two key datasets—log_in_attempts and employees—to identify suspicious behavior such as failed logins outside business hours, access from unknown locations, and machines needing updates.

Retrieve After-Hours Failed Login Attempts

SELECT *
FROM log_in_attempts
WHERE login_time > '18:00'
  AND (success = 0 OR success = FALSE);

Retrieve Login Attempts on Specific Dates

SELECT *
FROM log_in_attempts
WHERE login_date = '2022-05-09'
   OR login_date = '2022-05-08';

Retrieve Login Attempts Outside of Mexico

SELECT *
FROM log_in_attempts
WHERE country NOT LIKE 'MEX%'
  AND country NOT LIKE 'Mexico%';

Retrieve Employees in Marketing

SELECT *
FROM employees
WHERE department = 'Marketing'
  AND office LIKE 'East%';

Retrieve Employees in Finance or Sales

SELECT *
FROM employees
WHERE department = 'Finance'
   OR department = 'Sales';

Retrieve All Employees Not in IT

SELECT *
FROM employees
WHERE department != 'Information Technology';

Summary

This activity demonstrated my ability to apply SQL filtering techniques—using logical operators, date/time conditions, and pattern matching—to identify security-relevant anomalies in login and employee data.

Identifying Vulnerabilities for a Small Business

Project Description

I conducted a hands-on vulnerability assessment of a publicly accessible Linux server hosting a MySQL database for a fictional e-commerce organization. Using the NIST SP 800-30 framework, I analyzed threats, quantified likelihood and impact, and proposed remediation strategies to protect customer data and ensure service availability.

Scenario

As part of my cybersecurity learning journey, the organization’s database server—critical for business operations and storing customer data—has been online and publicly accessible for three years. It runs Linux, hosts MySQL, supports remote employee access worldwide, and uses SSL/TLS for data transmission. This exercise simulated a real-world assessment following NIST SP 800-30 Rev. 1.

System Description

The server features a high-performance CPU and 128 GB RAM, latest Linux OS, and MySQL. It communicates over IPv4 with internal servers, allows global remote access, and secures connections via SSL/TLS. No firewall restrictions exist for inbound database traffic.

Scope

This assessment focuses on the confidentiality, integrity, and availability of data on the publicly accessible database server. It excludes physical security and other IT infrastructure. The three-month evaluation followed NIST SP 800-30 guidelines.

Purpose

The database is essential for sales and marketing. A breach could lead to data theft, legal issues, reputational harm, and operational downtime. This report aims to identify vulnerabilities before exploitation and recommend controls to mitigate risk.

Risk Assessment

Threat Source Threat Event Likelihood Severity Risk (L × S)
Cybercriminals Exfiltrate customer data via SQL injection 3 3 9
Insider (employee) Misuse credentials to access restricted data 2 3 6
Competitor Launch DDoS attack to disrupt operations 2 2 4

Approach

Using NIST SP 800-30 Rev. 1, I scored threats based on actor capability and system exposure. Likelihood and severity ratings were estimated from threat actor profiles and potential business impact. Focus was on cyber-based vectors due to public accessibility.

Remediation Strategy

  • Implement Principle of Least Privilege to restrict data access by role.
  • Enforce Multi-Factor Authentication (MFA) for all database accounts.
  • Restrict server access behind a VPN or firewall ACLs.
  • Deploy continuous monitoring and regular vulnerability scans.

Importing and Parsing a Text File in a Security-Related Scenario

Project Description

I acted as a security professional at a fictional healthcare company, tasked with maintaining a list of employee IP addresses that are permitted to access a restricted subnetwork. This list, stored in allow_list.txt, must be regularly updated to remove any IPs identified in a separate remove_list of revoked addresses. The goal is to automate the removal of any IPs in the remove list from the allow list using Python.

Scenario

I simulate a real-world network admin task: given allow_list.txt containing authorized IPs and a predefined remove_list, write a Python script to filter out revoked addresses and update the file. This ensures only currently authorized IPs retain access.

Open & Read the File

with open("allow_list.txt", "r") as file:
    ip_addresses = file.read().split()

Iterate & Remove

remove_list = ["192.168.1.1", "10.0.0.5"]
for ip in remove_list:
    if ip in ip_addresses:
        ip_addresses.remove(ip)

Write the Updated List

with open("allow_list.txt", "w") as file:
    file.write("\n".join(ip_addresses))

Summary

This Python algorithm automates access control by: 1. Reading the existing allow-list file.
2. Parsing its contents into a list.
3. Removing any IPs found in a predefined revoke list.
4. Writing the cleaned list back to the file. The result is a self-updating allow-list that reflects current authorization policies.
