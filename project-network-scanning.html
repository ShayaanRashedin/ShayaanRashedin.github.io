Network Scanning & Reconnaissance — Nmap & OpenVAS

Tools: Kali Linux, Nmap, OpenVAS / GVM, VirtualBox (host-only network)

Objective: Identify exposed hosts and services in a controlled lab network and produce prioritized findings for follow-up exploitation or defensive hardening.

Environment & Setup

The lab runs in VirtualBox with the attacker (Kali) and target (Metasploitable2) on a host-only network (example: 192.168.56.0/24). Always snapshot targets before running noisy scans. The outputs below were captured from the Kali attacker terminal and the Metasploitable2 target.

Reconnaissance Workflow (phases)

Workflow: Host discovery → Port & service enumeration → Automated vulnerability scanning. Each phase includes the commands used and evidence (screenshots).

Phase 1 — Host discovery

Commands (Host discovery) 
ping 192.168.56.101

Confirms the host is reachable before scanning. (See Figure 1)

Phase 2 — Full TCP port discovery

Commands (Full TCP port scan) 
nmap -sS -p- -oN scans/full-port-scan.txt 192.168.56.101

Finds every open TCP port so we can enumerate the services running. (See Figures 2–3)

Phase 3 — Service & version enumeration

Commands (Service/version detection) 
nmap -sV -sC -oN scans/version-os-scan.txt 192.168.56.101

Identifies service names and versions (e.g., vsftpd 2.3.4, Apache 2.2.8), which map to CVEs and exploitability. (See Figures 4 & 5)

Auxiliary verification

Commands (HTTP headers & target interfaces) 
curl -I http://192.168.56.101:80
# target-side checks (on Metasploitable2)
ifconfig
ip addr

HTTP headers confirm server software; target interface output confirms IP addresses used in the lab. (See Figures 6–8)

Automated Vulnerability Scanning — OpenVAS / GVM

After manual enumeration, an OpenVAS scan (Full & Fast) was run to get prioritized findings and CVE references. I kept your existing XSL/XML rendering for the OpenVAS report intact (no changes to your OpenVAS XSL/XML integration).

If you want a direct link to the rendered OpenVAS report, use your existing `openvas-report.html` (which already reads openvas-report.xml with openvas.xsl).

Analysis & Findings

Nmap + OpenVAS identified legacy services and weak/default configurations (vsftpd backdoor indicators, UnrealIRCd, outdated PHP/Apache). Findings were prioritized by CVSS and exploit availability; these shaped the follow-up exploitation tests in a separate project.

Remediation & Defensive View

  • Patch or remove obsolete services; update Apache/PHP/tomcat to fixed releases.
  • Disable unused network-facing daemons (rsh, rexec, old FTP) when possible.
  • Replace default credentials and enforce strong password policies.
  • Apply network segmentation and host-based firewalls to limit exposure.
  • Feed scan results into your SIEM and build alerts around high CVSS or unexpected service discovery.

Reflection

This recon phase reinforced combining targeted Nmap scans with automated scans to form a prioritized attack surface map — a repeatable workflow for both red- and blue-team contexts.
